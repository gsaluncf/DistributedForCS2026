Week 7 – Replication: Why Copies Break Everything

If one copy of the data is good, more copies should be better — right? This week we learn why the answer is "it depends," and why the cost of keeping copies consistent is often the hardest problem in a distributed system.

Reading: Sections 7.1 through 7.6 (~40 pages). We will spend the most time on 7.1 (why replicate), 7.3 (client-centric consistency), and 7.6 (caching and replication in the web).

Two Reasons to Replicate

There are really only two reasons you would go through the trouble of maintaining multiple copies of the same data: reliability and performance.

Reliability is the simpler motivation. If you have three copies of a file and one server crashes, you switch to another. You can even detect corruption: if two out of three copies return the same value, you trust the majority. The more copies you have, the harder it is for a single failure to take you down.

Performance is about distance and load. Imagine a single server in Virginia serving users in Tokyo, São Paulo, and Berlin. Every request has to cross an ocean. That round-trip latency is physics — you cannot make light travel faster. But you can put a copy of the data on a server in each region. Now reads are fast because the data is nearby. And if a thousand users are all requesting the same data, you can spread that load across multiple servers instead of overwhelming one.

The basic idea: one origin, multiple replicas closer to users. Reads are fast — but keeping all copies in sync costs network traffic and time.

Both motivations sound obviously good. So what goes wrong?

The Fundamental Problem

The moment someone modifies the data on one copy, that copy is now different from all the others. To get back to a consistent state, the modification has to be carried out on every copy. Exactly when and how those modifications happen determines the price of replication. Everything else in this chapter is really about that one trade-off.

The Access-to-Update Ratio: A Real-World Intuition

Before we get into consistency models and protocols, there is one number that can tell you immediately whether replication is a good idea for a particular piece of data: the access-to-update ratio.

Here is the idea. Imagine a piece of data that is read N times per second but updated M times per second. Every update has to be propagated to every replica — that costs network bandwidth and time. If N is much larger than M (lots of reads, few updates), those propagation costs are spread across many useful reads. The investment pays off. But if N is close to M, or worse, M is larger than N, then you are constantly shipping updates to replicas that nobody is reading. You are spending more on keeping copies fresh than you are saving on faster reads.

This is not abstract. Think about data you interact with every day:

  • A Wikipedia article about World War II. Read millions of times a day, edited maybe a few times a week. Enormous ratio. Cache it on servers all over the world. The occasional edit is cheap to push out.
  • A stock ticker for a heavily traded company. The price updates many times per second. Even if thousands of people are watching, the data changes so fast that any cached copy is stale almost immediately. If you replicate it to ten edge servers, you are shipping ten updates per second to each one — and most of those updates will be superseded before anyone reads them.
  • Your profile picture on a social platform. You change it once every few months. It is loaded every time someone sees one of your posts — potentially thousands of times a day. Extremely high ratio. This is the kind of data that CDNs were born to cache.
  • A collaborative Google Doc being edited by five people. Every keystroke is an update, and every collaborator needs to see every change almost instantly. The ratio is roughly 1:1. Caching will not help here — you need a completely different synchronization strategy that keeps everyone's view of the document current in real time.

The simplest example is web caching. Your browser stores a local copy of a page for fast access. That cached copy has an excellent access-to-update ratio — you visit the page many times, and it rarely changes. But if the page does change on the server, your cached copy is now wrong. The server could notify every browser cache of every change, but tracking millions of caches is expensive. This is the core tension: caching makes reads fast but makes freshness hard.

What Does "Consistent" Even Mean?

Once you have multiple copies, you have to decide what "correct" means. If someone reads from Replica A and someone else reads from Replica B at the same time, do they have to see the same thing? Always? Sometimes? What if a user writes to one replica and then reads from another — do they have to see their own write?

The answer is: it depends on what consistency model you choose. A consistency model is a contract between the system and its users. It says: "here is what we guarantee about the relationship between writes and reads." Stronger guarantees mean more predictable behavior for users, but they cost more to implement because the replicas have to coordinate more tightly.

The key insight is that consistency is not a binary — it is a spectrum. At one end, every replica is always perfectly identical (very expensive). At the other end, replicas can be arbitrarily out of date (very cheap but confusing for users). Most real systems live somewhere in the middle.

The Strongest Guarantee (and Why It Is Expensive)

At the far right of the spectrum is tight consistency (also called synchronous replication). This means that when someone writes a value, the write is applied to every single copy as a single atomic operation before anything else can happen. No one can read any copy until all copies agree.

Think about what this requires. You post a photo. Before the app can even show you a confirmation, the system has to contact servers in Virginia, Frankfurt, Tokyo, and São Paulo, wait for all of them to acknowledge the write, and only then tell you it worked. If one server is slow or unreachable, everything waits. If the network between two data centers hiccups, all writes globally stall. You get perfect consistency, but you pay for it in latency and fragility. This is why almost no large-scale internet application uses tight consistency for everything.

Continuous Consistency: How Wrong Is Acceptable?

An interesting idea from the reading is conits (consistency units) — a way to specify exactly how much replicas are allowed to disagree. You can set bounds along three dimensions:

  • Numerical deviation: How far apart can the values be? (Useful for things like account balances where small discrepancies might be tolerable.)
  • Staleness: How old can a copy be? This is the one you encounter most often. When your browser shows a cached web page, it is accepting a staleness bound — a page that is five minutes old is probably fine; five days old is probably not.
  • Ordering deviation: How many writes can be "pending" at one replica that other replicas have not seen yet?

The practical value of conits is that they turn consistency from a yes/no question into a tunable parameter. Instead of "are the replicas consistent?" you ask "are the replicas consistent enough for this particular use case?"

Client-Centric Consistency: What One User Sees

There is a different way to think about consistency that turns out to be very practical. Instead of asking "do all replicas agree?" — which is expensive — you ask: "does this one user have a coherent experience as they move between replicas?"

This matters because in most real systems, a user does not stay connected to one server forever. A load balancer might route your first request to a server in Virginia and your second request to a server in Oregon. If those two servers have different versions of the data, your experience might be confusing even if the system as a whole is working fine.

There are four guarantees that matter here, and two of them come up constantly in practice:

Monotonic reads means that once you have seen a value, you never see an older one. Think about checking your bank balance. You see $500. You pull down to refresh. Now it shows $480 (a charge went through). You refresh again — and it shows $500 again. That third read is the problem. The value went backward. The second server you hit had an older copy of the data than the first one. A system that guarantees monotonic reads will never let this happen — it ensures that any server you are sent to has at least the version you already saw.

Read-your-writes means that after you write something, your own subsequent reads will reflect that write. You update your display name from "alice99" to "Alice Chen." You reload the page. It still says "alice99." What happened? Your write went to one server, but your read was routed to a different server that had not received the update yet. This is surprisingly common — it happens any time a load balancer sends consecutive requests to different replicas.

The other two client-centric guarantees are monotonic writes (your writes are applied everywhere in the order you issued them — so no replica ever shows your correction without the original post) and writes-follow-reads (if you read an article and then post a reply, no replica will store your reply without also storing the article you replied to).

The important thing to notice about all four of these guarantees is that they are about the experience of a single user moving between replicas. They do not require every replica in the system to agree with every other replica at all times. That is what makes them much cheaper to implement than tight consistency — and why so many real systems rely on them.

Replication in the Real Web: CDNs and Edge Caching

Let us take everything we have discussed — replication, the access-to-update ratio, consistency trade-offs — and see how it plays out in one of the most important real-world distributed systems: content delivery networks (CDNs).

The Problem CDNs Solve

Imagine you run a popular news website. Your database and application servers live in a single data center in Virginia. A reader in Berlin requests your homepage. That request has to travel across the Atlantic, hit your server, generate a response (which might require database queries), and travel all the way back. Round-trip latency alone might be 100–150 milliseconds, plus however long the server takes to process the request. Multiply that by millions of users worldwide, and you have a performance problem that no amount of server optimization can fully solve — because the bottleneck is the speed of light across an ocean.

The CDN solution is replication. You deploy edge servers in cities around the world — Berlin, Tokyo, São Paulo, Sydney. These edge servers hold copies of your content. When the reader in Berlin requests your homepage, a CDN routes them to the nearest edge server instead of your origin. The page loads fast because the data is close.

But here is the question that should immediately come to mind after this week's reading: what exactly do you put on those edge servers, and how do you keep it fresh?

The reading describes three strategies, and each one makes a fundamentally different trade-off between how much the edge server knows, how much storage it uses, and how hard it is to keep consistent with the origin.

Strategy 1: Full Replication

The most straightforward approach: copy the entire origin database to every edge server. Every query can be answered locally because the edge has all the data. Reads are fast. Complex queries that need to join multiple tables work fine because the full dataset is right there.

The catch is updates. Every time a row changes in the origin database, that change has to be propagated across a wide-area network to every edge server. If your site publishes ten articles a day, this is fine — the update traffic is minimal. But if your site processes thousands of user-generated comments per minute, you are constantly shipping updates across oceans to dozens of edge servers, many of which might not even have users requesting that data. The access-to-update ratio is the deciding factor.

Strategy 2: Content-Aware Caching

Instead of copying everything, the edge server only stores data that users actually request — but it stores it in a structured way. The edge maintains a small local database organized around query templates: predefined patterns like "get user profile where id = ?" or "get top 10 posts where category = ?".

When a request arrives, the edge checks whether it matches a template and whether the local database has the data needed to answer it. This is called a query containment check — essentially asking "can I answer this without calling home?" If yes, it responds instantly. If not, it forwards the query to the origin, stores the response locally, and returns it to the user. Next time someone asks a similar question, the edge can answer it directly.

The trade-off: this is much more storage-efficient than full replication because the edge only holds data people actually care about. But it introduces two new costs. First, the origin server has to track what each edge has cached so it can send invalidation messages when the underlying data changes. ("Hey Berlin, the article you cached was just edited — throw out your copy.") Second, the edge server needs real computational power because it is not just serving static files — it is running database queries.

Strategy 3: Content-Blind Caching

This is the simplest approach, and it is more common than you might think. The edge server does not understand the data at all. It does not have templates. It does not run a database. All it does is remember the exact queries it has seen before.

When a request comes in, the edge computes a hash of the query string. It checks its cache: "have I seen this exact query before?" If yes, it returns the stored result. If no, it forwards the query to the origin, stores the result keyed by its hash, and returns it to the user. Next time someone sends the exact same query, the edge can answer instantly.

The advantage is that the edge needs almost no intelligence — it is just a key-value store of query hashes to results. No database engine, no query processing. This is cheap to operate.

The disadvantages are real, though. First, storage can be wasteful. Two different queries might return overlapping data, but the edge stores both results separately because it does not understand their contents. Second, invalidation is hard. When the origin data changes, the origin has to figure out which cached queries might be affected — and without understanding the query structure, that is a guessing game. Third, even small changes to a query (adding a parameter, changing capitalization) make it a "new" query in the edge's eyes, so it misses the cache even though a nearly identical query was answered moments ago.

The Pattern to Notice

Each strategy trades edge intelligence against consistency complexity. Full replication gives the edge everything — any query works, no forwarding needed — but keeping all those copies consistent is expensive. Content-aware caching is smarter about what to store, so updates affect fewer cached items, but the origin has to carefully track who has what. Content-blind caching asks the least of the edge server, but it is the most wasteful in storage and the hardest to invalidate precisely because the origin cannot tell which cached results are affected by a given change. There is no free lunch — you are always choosing where to pay the cost.

Connecting to Our 2032 System

In our system, each site hosts identities and publishes events to other sites. Think of each site as an origin server for its own users' data. When a partner site wants to display information about those users — a profile card, a display name, a follower count — it is acting as an edge server. It needs a copy of the data, and it has to decide how to get one.

The three CDN strategies map directly to choices our system has to make:

  • Full replication: Site A maintains a complete copy of Site B's user directory. Any lookup is instant. But every time someone on Site B updates their profile, Site B has to push that change to every partner site. If a hundred sites replicate your directory, that is a hundred update messages for every profile change.
  • Content-aware caching: Site A caches profiles it has been asked about, organized by the kinds of lookups it supports. More targeted — you only cache what your users actually need. But Site B has to know what Site A cached, so it can send invalidation messages when profiles change.
  • Content-blind caching: Site A hashes each profile request and caches the result. Simplest approach. But if a user changes their display name on Site B, Site A has no way to know which cached entries are now wrong. It just has to wait for them to expire.

And the access-to-update ratio guides the choice. User profiles change rarely and are read often — cache aggressively, any strategy works. Activity feeds change constantly — caching is harder, staleness is visible to users almost immediately, and the "right" choice depends on how much staleness your users will tolerate.

Part 1: Seeing the Problems

Part 1 builds the simplest possible replication system: one origin server and two edge replicas. All writes go to the origin, which propagates them to the edges in background threads — but with a configurable delay that simulates real network latency. This is primary-based replication, the kind of setup you would see in a basic CDN.

You run four experiments against this system. The first uses zero delay (the ideal world where all replicas agree instantly). The other three introduce a 2–3 second delay and show you exactly what breaks:

  • Stale reads: You write a value to the origin, then immediately read from an edge. The edge returns None because the propagation thread has not finished yet. This is the basic staleness problem — the edge is serving outdated data during the propagation window.
  • Read-your-writes violation: Alice updates her display name on the origin. A load balancer routes her next request to an edge that has not received the update. She reloads the page and does not see her own change. The system accepted her write, but from her perspective it vanished.
  • Monotonic reads violation: A follower count is updated from 1000 to 1050 on the origin. A user reads from the origin (sees 1050), then refreshes and gets routed to an edge that still has 1000. The value went backward.

In all three cases, the system is working exactly as designed. There are no bugs. The "problems" are consequences of the consistency model — or the lack of one. Whether they matter depends on the application. A stale follower count is annoying. A stale bank balance is unacceptable. That judgment call is the point.

Part 2: Three Caching Strategies

Part 1 showed that propagation delay causes consistency problems. Part 2 asks a different question: what data should the edge servers have in the first place? It implements all three CDN caching strategies from the lesson as working Python classes, running them against the same origin server with the same data — so you can see how they compare head to head.

  • Full replication (FullReplicationEdge): The edge copies the entire origin database when it is created. Every query is answered locally — no forwarding, no cache misses. You write to the origin, read from the edge (still stale), then call propagate_update() to push the change, and read again (now fresh). You can see the cost: every update at the origin requires an explicit push to every edge.
  • Content-aware caching (ContentAwareEdge): The edge starts empty and learns what data to hold based on incoming queries. It understands the structure of the data (query templates like "get article by ID"). First queries are cache misses that get forwarded to the origin and then cached. Repeated queries are hits. When the origin updates a record, it sends an invalidate() message. The next query re-fetches from the origin. You can see both the efficiency (only caching what users request) and the coordination cost (the origin must track what each edge has).
  • Content-blind caching (ContentBlindEdge): The edge hashes each query string with SHA-256. Same exact query = cache hit. Even a tiny variation — a capital letter, a trailing space — produces a different hash and is treated as a brand new query. You can see this happening in the output. Invalidation is handled by a TTL (time-to-live): the cell actually waits for the TTL to expire, so you watch the edge serve stale data until the timer runs out and forces a fresh fetch.

The notebook ends Part 2 with a comparison table that prints all three strategies side by side across several dimensions: whether the edge starts with data, whether it understands the data, how invalidation works, storage cost, and compute requirements.

Part 3: Fixing Read-Your-Writes

Part 3 takes the broken system from Part 1 and fixes one specific problem: the read-your-writes violation. It implements sticky sessions — when a user writes to the origin, the routing layer remembers that user and routes their subsequent reads to the origin (instead of a possibly-stale edge) for the duration of the propagation window.

You run the same scenario — Alice updates her display name, then immediately reads — and this time she sees her own change because the system routed her back to the origin. Meanwhile, Bob (who did not write anything) still gets routed to the edge, where he might see stale data — but that is expected for non-writers.

The notebook explains the trade-offs honestly. Sticky sessions fix read-your-writes, but they create uneven load (users who write frequently always go to the origin), reduced fault tolerance (if the origin goes down, sticky users cannot be rerouted), and the routing layer now needs per-user state. The wrap-up also describes alternative fixes — write-through to edges, read-repair, and client-side version tracking — so you can see that sticky sessions are one tool among several, not the only answer.
