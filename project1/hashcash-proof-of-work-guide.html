From Choking to HashCash: Proof-of-Work as Message Rate-Limiting

We have solved who I will serve (choking) and who I trust (reputation). But we have not solved a more fundamental problem: how expensive is it to talk?

Choking punishes free-riders after the fact. HashCash makes free-riding expensive before the fact. This is the difference between a bouncer who kicks you out and a cover charge that keeps you honest at the door.

The AdFlow Problem: Cheap Messages, Expensive Consequences

Think about our current AdFlow network. Every node can send messages to the SQS queues of every other node. There is nothing stopping a malicious node from doing this:

malicious_node floods the network:

  AUDIT_RESULT(show:midnight-run, count=999999)  # Fake audit
  AUDIT_RESULT(show:midnight-run, count=999999)  # Same fake, again
  AUDIT_RESULT(show:midnight-run, count=999999)  # And again
  ... 10,000 more in the next second ...

  GOSSIP_ANNOUNCE(node_id=evil-node-1)  # Fake peer
  GOSSIP_ANNOUNCE(node_id=evil-node-2)  # Another fake
  GOSSIP_ANNOUNCE(node_id=evil-node-3)  # Sybil attack
  ... 50,000 fake nodes announced ...

Right now, sending an SQS message costs the sender nothing. JSON serialization is fast. Network calls are cheap. A single laptop can flood the entire AdFlow network with garbage messages in seconds.

What Choking Does Not Solve

Our existing choking algorithm is reactive. It tracks contributions over time and punishes peers who take without giving. But it has blind spots:

Attack Does Choking Help? Why Not?
Message flooding No Choking decides who I serve, not who can talk to me. Messages arrive before I decide to choke.
Sybil attacks No Creating fake identities is free. An attacker generates 100 node IDs and gossips them all.
Fake audit results Sort of Reputation will eventually flag the liar, but the damage is done by the time choking kicks in.
Resource exhaustion No Even choked peers can fill my SQS queue, costing me polling time and CPU.

The Core Insight

Choking is a receiver-side defense. It controls who I serve. But it does not control who talks to me, or how often. We need a sender-side cost - something that makes it expensive to produce messages, not just expensive to get a response.

Enter HashCash: Making Messages Expensive to Forge

Adam Back, 1997

HashCash was originally invented to fight email spam. The idea is deceptively simple:

The Big Idea

Before you send a message, you must burn CPU cycles to produce a proof that you spent real computational work. The recipient can verify this proof instantly. Sending one email? Trivial cost. Sending a million spam emails? Bankruptingly expensive.

How It Works: The Hash Puzzle

A cryptographic hash function like SHA-256 takes any input and produces a fixed-length output that looks random. Crucially, there is no shortcut - you cannot predict the output without actually computing the hash.

import hashlib

message = "AUDIT_RESULT:show=midnight-run:count=1000:sender=hugo"

# Try nonce = 0
attempt = message + ":nonce=0"
hash_0 = hashlib.sha256(attempt.encode()).hexdigest()
# hash_0 = "a4f3e2c1..."  (random-looking, no leading zeros)

# Try nonce = 1
attempt = message + ":nonce=1"
hash_1 = hashlib.sha256(attempt.encode()).hexdigest()
# hash_1 = "8b29d0f7..."  (still no leading zeros)

# Try nonce = 2
attempt = message + ":nonce=2"
hash_2 = hashlib.sha256(attempt.encode()).hexdigest()
# hash_2 = "f17a90bc..."  (nope)

# ... keep trying ...

# Try nonce = 84,231
attempt = message + ":nonce=84231"
hash_n = hashlib.sha256(attempt.encode()).hexdigest()
# hash_n = "00000a3f..."  FOUND IT! Starts with 5 zeros!

The rule: Your message is only valid if its hash starts with a certain number of leading zeros. Finding such a hash requires brute-force guessing of the nonce value. There is no shortcut.

The Asymmetry: Hard to Produce, Easy to Verify

This is the key property that makes HashCash useful:

Operation Cost Who Pays
Mining the proof (finding the nonce) Thousands to millions of hash attempts Sender
Verifying the proof (checking the hash) Exactly 1 hash computation Receiver

A legitimate sender producing 1 message pays a small cost, maybe half a second of CPU time. A spammer trying to produce 10,000 messages pays 10,000 times that cost - hours of computation for what should be a trivial script.

The Difficulty Dial

The number of leading zeros is the difficulty parameter. More zeros means more required work:

Leading Zeros Avg. Attempts Needed Approx. Time (on a laptop) Use Case
1 ("0...") 16 Instant Trivial proof, just forces the sender to hash at all
4 ("0000...") 65,536 Less than a second Our AdFlow messages
5 ("00000...") 1,048,576 A few seconds Audit results (higher-value messages)
8 ("00000000...") 4,294,967,296 Minutes to hours Bitcoin block production (roughly, before 2012)
19+ (current Bitcoin) Trillions 10 minutes (entire network combined) Global consensus

Why Hex Zeros?

Each hex character represents 4 bits. A leading zero in hex means 4 leading zero bits. So 4 hex zeros = 16 zero bits. Each additional zero bit doubles the average work needed - exactly like adding one more digit to a combination lock.

HashCash in AdFlow: Stamped Messages

Here is how HashCash applies to our P2P network. Every message sent through SQS must include a proof-of-work stamp. No stamp, no processing.

The Message Format (Before and After)

BEFORE (current - no proof required):
{
    "type": "AUDIT_RESULT",
    "sender": "hugo",
    "content_id": "midnight-run",
    "agreed_count": 1000,
    "confidence": 0.85,
    "timestamp": 1711900000
}

AFTER (with HashCash stamp):
{
    "type": "AUDIT_RESULT",
    "sender": "hugo",
    "content_id": "midnight-run",
    "agreed_count": 1000,
    "confidence": 0.85,
    "timestamp": 1711900000,
    "pow": {
        "nonce": 84231,
        "difficulty": 4,
        "hash": "0000a3f7e2b1c..."
    }
}

Sender Side: Mining the Stamp

import hashlib, json

def mine_stamp(message_body: dict, difficulty: int = 4) -> dict:
    """
    Find a nonce that makes the message hash start with `difficulty` zeros.
    Returns the proof-of-work dict to attach to the message.
    """
    # Canonical serialization (sorted keys, no whitespace)
    canonical = json.dumps(message_body, sort_keys=True, separators=(",", ":"))

    nonce = 0
    target_prefix = "0" * difficulty

    while True:
        attempt = f"{canonical}:{nonce}"
        h = hashlib.sha256(attempt.encode()).hexdigest()
        if h.startswith(target_prefix):
            return {"nonce": nonce, "difficulty": difficulty, "hash": h}
        nonce += 1

Receiver Side: Verifying the Stamp

def verify_stamp(message_body: dict, pow_data: dict) -> bool:
    """
    Verify a proof-of-work stamp. One hash computation.
    Returns True if the proof is valid.
    """
    canonical = json.dumps(message_body, sort_keys=True, separators=(",", ":"))
    attempt = f"{canonical}:{pow_data['nonce']}"
    h = hashlib.sha256(attempt.encode()).hexdigest()

    # Check: does the hash actually start with enough zeros?
    expected_prefix = "0" * pow_data["difficulty"]
    return h.startswith(expected_prefix) and h == pow_data["hash"]

The Asymmetry in Action

Sender (hugo): Ran ~65,000 hash attempts over half a second to mine the stamp.

Receiver (bot-charlie): Ran exactly 1 hash to verify. Took microseconds.

For one message, this is barely noticeable. For 10,000 spam messages, the spammer needs 9+ hours of CPU time.

Two Layers of Defense

Choking and HashCash are complementary. They defend different surfaces at different times:

Messages must pass both layers: the proof-of-work gate (sender cost) and the choking/reputation gate (receiver discretion).

Variable Difficulty: Not All Messages Are Equal

In AdFlow, different message types have different consequences. A gossip announcement is low-stakes. An audit result that triggers a payment is high-stakes. The difficulty should reflect the stakes:

Message Type Difficulty Rationale
GOSSIP_ANNOUNCE 2 Low-stakes. Easy to disprove fake peers through heartbeat.
HEARTBEAT 2 High frequency, low cost. Must not slow down liveness detection.
VIEW_REPORT 4 Ad-view claims directly affect revenue. Moderate cost to forge.
AUDIT_RESULT 5 Triggers payments. High-value. Worth spending a few seconds per stamp.
ELECTION 4 Rare event. Prevents an attacker from flooding election messages.

Design Decision

The network can agree on difficulty levels per message type through configuration. This is similar to how Bitcoin adjusts its block difficulty every 2,016 blocks - except we set it statically based on message importance rather than dynamically based on hashrate.

From Stamps to Chains: The Road to Bitcoin

HashCash stamps individual messages. Each stamp is independent - it proves work was done for that message, and then it is forgotten. But what if we started linking stamps together?

HashCash (what we are doing now):
  msg1 + nonce  -->  hash = "0000..."     (standalone stamp)
  msg2 + nonce  -->  hash = "0000..."     (standalone stamp)
  msg3 + nonce  -->  hash = "0000..."     (standalone stamp)

What if we chained them?
  msg1 + nonce                  -->  hash1 = "0000abc..."
  msg2 + nonce + hash1          -->  hash2 = "0000def..."   (includes previous hash!)
  msg3 + nonce + hash2          -->  hash3 = "0000789..."   (includes previous hash!)

Now changing msg1 would BREAK hash2 AND hash3.

This is exactly what Bitcoin does. A blockchain is nothing more than a chain of HashCash stamps where each stamp commits to the entire history before it. We will explore this connection in detail when we cover distributed consensus.

The Evolution

Choking (BitTorrent, 2001) "I will stop serving you if you do not contribute." Receiver-side punishment.
HashCash (Back, 1997) "You must prove you spent CPU time before I read your message." Sender-side cost.
Bitcoin (Nakamoto, 2008) "Chain the proofs together so the entire transaction history is protected by cumulative work." Consensus from proof-of-work.

Each step takes the same fundamental mechanism - hashing as a source of computational cost - and applies it to a bigger problem.

Properties and Tradeoffs

Property HashCash Notes
Decentralized Yes No central authority needed. Any node can verify any stamp.
Stateless verification Yes Receiver needs no history. Each stamp is self-contained proof.
Replay protection Partial The stamp is tied to the message content. Replaying the same message+stamp is valid unless the receiver tracks seen hashes.
Fairness Imperfect A machine with a fast CPU mines stamps faster. This favors wealthy attackers.
Energy cost Low (per message) At difficulty 4, the energy is negligible. At Bitcoin-scale difficulty, it is enormous.
Sybil resistance Yes Creating 1,000 fake identities requires 1,000 stamps. Identity is no longer free.

The Fairness Problem (Why Bitcoin Invented Mining Pools)

HashCash has a structural weakness: it punishes the honest but slow. If a student's laptop takes 3 seconds to mine a stamp while a bot's server does it in 0.1 seconds, the system works well - the cost is proportional to the resources available. But what about an attacker with access to cloud GPUs? They could mine stamps orders of magnitude faster than legitimate participants.

This is acceptable for spam prevention (even fast hardware cannot send millions of stamped messages cheaply). It is not acceptable if proof-of-work is used for consensus - and this tension is exactly what led Bitcoin from "one stamp per message" to "one stamp per block, compete globally."

Where It Fits: Integration with Our Codebase

HashCash is not a separate algorithm module like gossip or choking. It is a cross-cutting concern that touches the message layer itself. There are two integration points:

node.py  _send_to_peer()
  |
  |  1. Build message dict
  |  2. Mine PoW stamp        <-- NEW: stamp before sending
  |  3. Attach pow field
  |  4. Send to SQS
  |
  
node.py  _process_message()
  |
  |  1. Receive from SQS
  |  2. Verify PoW stamp      <-- NEW: reject if invalid
  |  3. If invalid, drop + log
  |  4. If valid, route to handler

The implementation can live in a utility module:

algorithms/
  gossip.py        peer discovery
  heartbeat.py     liveness detection
  choking.py       bandwidth allocation (receiver-side)
  reputation.py    trust scoring
  election.py      leader election
  hashcash.py      NEW: proof-of-work stamps (sender-side)

Discussion: From Rate-Limiting to Consensus

Think About

  • Choking vs. HashCash: We said choking is "reactive" and HashCash is "proactive." Can you think of a real-world analogy? (Hint: think about highways and toll roads.)
  • The Sybil connection: In our lab, creating a new node just means picking a new string as a node_id. If every GOSSIP_ANNOUNCE requires a difficulty-4 stamp, how does this change the cost of a Sybil attack? What if difficulty is set to 6?
  • Timestamp freshness: If we include the timestamp in the hash input, what happens when a valid stamped message is replayed 10 minutes later? How would you add replay protection without needing a central server?
  • The Bitcoin leap: HashCash stamps are per-message and independent. Bitcoin chains them and makes the entire network compete for one stamp every 10 minutes. What changes about the system properties when you go from "everyone mines their own stamps" to "everyone competes for one shared stamp"?
  • Energy: At difficulty 4, our AdFlow HashCash burns negligible energy. Bitcoin currently uses as much electricity as some countries. At what point does proof-of-work become socially irresponsible? Is there a threshold?

Complements Choking (receiver-side defense). HashCash + choking = defense at both ends.
