Step 5: Leader Election and Proof-of-Work

Your node can gossip, heartbeat, choke, audit, and track reputation. Two problems remain: who processes payments? and how do we stop spam?

This guide walks you through integrating two new systems into your node: Leader Election (selecting a single Payment Server) and HashCash Proof-of-Work (making messages expensive to forge).

Before You Start

Pull the latest code from the repo. You should see these new files:

  • algorithms/hashcash.py - Proof-of-work stamp library
  • algorithms/election.py - Leader election module
  • election_hashcash_sample.py - Runnable demo (start here!)

And these updated files:

  • protocol.py - 4 new message types added
  • config.py - 3 bots + new MESSAGE_TYPES

Part A: The Two Problems

Problem 1: Who Pays the Creator?

When an audit concludes, someone needs to issue a PAYMENT to the content creator. If all three bots issue payments independently, the creator gets paid three times. If none of them do, the creator gets nothing.

We need exactly one bot to act as the Payment Server at any given time. If that bot crashes, another must take over. This is the leader election problem.

Problem 2: Messages Are Free to Send

Right now, sending a message costs nothing. A malicious node can flood 10,000 fake AUDIT_RESULTs in seconds. Choking is reactive (receiver-side). We need a sender-side cost -- something that makes producing messages expensive. This is HashCash proof-of-work.

Part B: Four New Message Types

The protocol now has 12 message types (up from 8). Here are the new four:

Message Sent By Purpose
ELECTION Bot starting election "I want to be leader. Here is my reputation."
ELECTION_OK Higher-ranked bot "I outrank you. Back off, I will handle this."
COORDINATOR Election winner "I won. I am the Payment Server for term T."
PAYMENT Active leader only "Payment issued for content X based on audit Y."

The builders are already in protocol.py:

protocol.election(sender, term, reputation)        # ELECTION
protocol.election_ok(sender, term, reputation)     # ELECTION_OK
protocol.coordinator(sender, term)                 # COORDINATOR
protocol.payment(sender, content_id, agreed_count, # PAYMENT
                 amount, audit_ref="")

Part C: Adding Proof-of-Work to Your Node

The bots now verify incoming PoW stamps on VIEW_EVENT, AUDIT_RESULT, ELECTION, and PAYMENT messages. If your message lacks a valid stamp, the bot silently drops it. This means your node must stamp these messages before sending.

Step C.1: Add the import

At the top of my_node.py, add:

from algorithms.hashcash import stamp_message, verify_message

Step C.2: Create a helper method

Add this method to your MyNode class. It stamps a message with proof-of-work before sending it:

def _stamp_and_send(self, target_id, msg):
    """Stamp a message with PoW and send it."""
    stamped = stamp_message(dict(msg))  # copy so we don't mutate original
    if self.transport.send(target_id, stamped):
        self.stats["messages_sent"] += 1
        return True
    return False

Step C.3: Use it when sending VIEW_EVENT and AUDIT_RESULT

Everywhere you currently do self.transport.send(peer, msg) for a VIEW_EVENT or AUDIT_RESULT, replace it with:

# BEFORE (no proof-of-work):
self.transport.send(peer, msg)

# AFTER (with proof-of-work):
self._stamp_and_send(peer, msg)

Which messages need stamps?

Must stamp: VIEW_EVENT, AUDIT_RESULT (bots will drop these without PoW)

Do not need stamps: HELLO, PEER_LIST, PING, PONG (bots accept these without PoW)

Step C.5: Updated _do_publish example

Here is a complete _do_publish with PoW:

def _do_publish(self):
    """Publish a VIEW_EVENT with proof-of-work stamp."""
    content_id = random.choice(self.content_catalog)

    # Increment your local count
    if content_id not in self.local_counts:
        self.local_counts[content_id] = 0
    self.local_counts[content_id] += random.randint(80, 120)

    msg = protocol.view_event(
        sender=self.node_id,
        event_id=f"{self.node_id}-{uuid.uuid4().hex[:8]}",
        content_id=content_id,
        count=self.local_counts[content_id],
    )

    # Send with PoW stamp to all alive peers
    alive = self.heartbeat.get_alive_peers()
    for peer_id in alive:
        self._stamp_and_send(peer_id, msg)

    self._log(f"Published {content_id} count={self.local_counts[content_id]} "
              f"to {len(alive)} peers [PoW stamped]")

Part D: Handling Leader Election Messages

The bots run leader election among themselves. Your student node does not need to participate in the election algorithm, but you must handle the messages so you know who the Payment Server is.

Step D.1: Add state tracking

In your __init__, add:

def __init__(self, node_id, verbose=False):
    super().__init__(node_id, verbose)

    # ... your existing state ...

    # Election tracking
    self.current_payment_server = None   # who is the leader?
    self.current_term = 0                # election epoch
    self.payments_received = []          # log of payments

Step D.2: Add the four new handlers

def _handle_election(self, msg):
    """A bot is starting an election. Log it."""
    sender = msg["sender"]
    term = msg.get("term", 0)
    rep = msg.get("reputation", 0.0)
    self._log(f"[ELECTION] {sender} started election "
              f"(term={term}, rep={rep:.2f})")

def _handle_election_ok(self, msg):
    """A higher-ranked bot responded. Log it."""
    sender = msg["sender"]
    self._log(f"[ELECTION] {sender} sent ELECTION_OK "
              f"(outranks the challenger)")

def _handle_coordinator(self, msg):
    """A bot won the election. Track the new leader."""
    sender = msg["sender"]
    term = msg.get("term", 0)

    if term >= self.current_term:
        self.current_payment_server = sender
        self.current_term = term
        self._log(f"[ELECTION] New Payment Server: {sender} "
                  f"(term={term})")
    else:
        self._log(f"[ELECTION] Ignored stale COORDINATOR from "
                  f"{sender} (term={term} < {self.current_term})")

def _handle_payment(self, msg):
    """The Payment Server issued a payment. Log it."""
    sender = msg["sender"]
    content_id = msg.get("content_id", "?")
    amount = msg.get("amount", 0.0)
    agreed = msg.get("agreed_count", 0)

    self.payments_received.append({
        "from": sender,
        "content_id": content_id,
        "amount": amount,
        "agreed_count": agreed,
    })

    self._log(f"[PAYMENT] {sender} paid ${amount:.4f} for "
              f"{content_id} ({agreed} verified views)")

Step D.3: That is it -- they are already wired

The updated node.py base class already routes the 4 new message types to your handlers automatically. You do not need to override handle_message(). Just add the four handler methods from Step D.2 to your MyNode class and they will be called when the bots send these messages to you.

Part E: How the Election Works (For Your Understanding)

The bots use a Reputation-Weighted Bully Algorithm. Here is the sequence:

Step What Happens Algorithm Detail
1 Heartbeat detects leader is dead No PONG from leader for 90 seconds
2 A bot sends ELECTION to higher-reputation bots Bully: only challenges bots that outrank you
3 Higher-rep bots respond with ELECTION_OK Tie-break: higher node ID wins if reps are equal
4 Highest-rep bot gets no OK, declares itself leader 15-second timeout, then self-promote
5 Winner broadcasts COORDINATOR to all peers Includes term number to prevent stale overwrites
6 Only the leader processes audits into PAYMENTs Standby bots log audits but do not pay

Part F: PoW Difficulty Levels

Not all messages are equal. Higher-stakes messages require more proof-of-work:

Message Type Difficulty Avg Attempts Bot Verifies?
HELLO, PING, PONG, PEER_LIST 2 ~256 No (exempt)
VIEW_EVENT, ELECTION 3 ~4,096 Yes
AUDIT_RESULT, PAYMENT 4 ~65,536 Yes

Key Takeaway

stamp_message(msg) automatically picks the right difficulty based on the message type. You do not need to specify it manually.

Part G: Updated print_status

Update your status output to show the election and payment state:

def print_status(self):
    super().print_status()
    print(f"  Payment Server: {self.current_payment_server or 'unknown'}")
    print(f"  Election term:  {self.current_term}")
    print(f"  Payments seen:  {len(self.payments_received)}")
    for p in self.payments_received[-3:]:  # last 3
        print(f"    ${p['amount']:.4f} for {p['content_id']} "
              f"from {p['from']}")

Summary

Problem 1 Multiple bots, only one should pay. Solution: elect a leader.
Problem 2 Messages are free. Solution: attach proof-of-work stamps.
Algorithm Reputation-Weighted Bully: highest-reputation bot wins the election.
PoW SHA-256 HashCash. Find nonce so hash starts with N zeros. Hard to produce, instant to verify.
New messages ELECTION, ELECTION_OK, COORDINATOR, PAYMENT
Your job Stamp outgoing VIEW_EVENT/AUDIT_RESULT. Handle COORDINATOR and PAYMENT.
