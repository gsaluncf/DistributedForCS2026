Step 5: Leader Election and Proof-of-Work

Your node can gossip, heartbeat, choke, audit, and track reputation. Two problems remain: who processes payments? and how do we stop spam? This guide walks you through integrating Leader Election and HashCash Proof-of-Work into your node.

Before You Start

Pull the latest code. You need these files:

  • algorithms/hashcash.py - Proof-of-work stamp library
  • algorithms/election.py - Leader election module
  • protocol.py - 12 message types (4 are new)
  • election_hashcash_sample.py - Runnable demo (try this first!)

Part A: The Two Problems

Problem 1: Who Pays the Creator?

When an audit concludes, someone needs to issue a PAYMENT to the content creator. If all three bots pay independently, the creator gets triple-paid. We need exactly one bot acting as the Payment Server at any time. If that bot crashes, another takes over. This is the leader election problem.

Problem 2: Messages Are Free to Send

A malicious node can flood 10,000 fake AUDIT_RESULTs for free. Choking is reactive (receiver-side). We need a sender-side cost that makes producing messages computationally expensive. This is HashCash proof-of-work.

Part C: Adding Proof-of-Work to Your Node

Bots verify PoW stamps on VIEW_EVENT, AUDIT_RESULT, ELECTION, COORDINATOR, and PAYMENT messages. If your message lacks a valid stamp, the bot drops it silently.

C.1: Add the import

from algorithms.hashcash import stamp_message, verify_message

C.2: Create a helper method

def _stamp_and_send(self, target_id, msg):
    """Stamp a message with PoW and send it."""
    stamped = stamp_message(dict(msg))
    if self.transport.send(target_id, stamped):
        self.stats["messages_sent"] += 1
        return True
    return False

C.3: Use it when sending VIEW_EVENT and AUDIT_RESULT

# Instead of:
self.transport.send(peer, msg)

# Use:
self._stamp_and_send(peer, msg)

Which messages need stamps?

Must stamp: VIEW_EVENT, AUDIT_RESULT (bots drop these without PoW)

No stamp needed: HELLO, PEER_LIST, PING, PONG (bots accept these without PoW)

def _do_publish(self):
    content_id = random.choice(self.content_catalog)

    if content_id not in self.local_counts:
        self.local_counts[content_id] = 0
    self.local_counts[content_id] += random.randint(80, 120)

    msg = protocol.view_event(
        sender=self.node_id,
        event_id=f"{self.node_id}-{uuid.uuid4().hex[:8]}",
        content_id=content_id,
        count=self.local_counts[content_id],
    )

    for peer_id in self.heartbeat.get_alive_peers():
        self._stamp_and_send(peer_id, msg)

Part D: Handling Leader Election Messages

The bots run elections among themselves using a Reputation-Weighted Bully Algorithm. The bot with the highest reputation wins and becomes the Payment Server. The COORDINATOR message is signed with the winner's reputation -- receiving bots reject it if they outrank the claimed leader, triggering a counter-election until the true highest-rep bot wins.

Your student node observes this process. You must handle the messages so you know who the current Payment Server is.

D.1: Add state tracking in __init__

# Election tracking
self.current_payment_server = None   # who is the leader?
self.current_term = 0                # election epoch
self.payments_received = []          # log of payments

D.2: Add the four handlers

def _handle_election(self, msg):
    """A bot is starting an election. Log it."""
    sender = msg["sender"]
    term = msg.get("term", 0)
    rep = msg.get("reputation", 0.0)
    self._log(f"[ELECTION] {sender} started election "
              f"(term={term}, rep={rep:.2f})")

def _handle_election_ok(self, msg):
    """A higher-ranked bot responded. Log it."""
    sender = msg["sender"]
    self._log(f"[ELECTION] {sender} sent ELECTION_OK "
              f"(outranks the challenger)")

def _handle_coordinator(self, msg):
    """A bot won the election. Track the new leader."""
    sender = msg["sender"]
    term = msg.get("term", 0)
    rep = msg.get("reputation", 0.0)

    if term >= self.current_term:
        self.current_payment_server = sender
        self.current_term = term
        self._log(f"[ELECTION] Payment Server: {sender} "
                  f"(term={term}, rep={rep:.3f})")
    else:
        self._log(f"[ELECTION] Ignored stale COORDINATOR "
                  f"from {sender} (term={term})")

def _handle_payment(self, msg):
    """The Payment Server issued a payment. Log it."""
    sender = msg["sender"]
    content_id = msg.get("content_id", "?")
    amount = msg.get("amount", 0.0)
    agreed = msg.get("agreed_count", 0)

    self.payments_received.append({
        "from": sender,
        "content_id": content_id,
        "amount": amount,
        "agreed_count": agreed,
    })

    self._log(f"[PAYMENT] {sender} paid ${amount:.4f} for "
              f"{content_id} ({agreed} verified views)")

The base class node.py already routes these message types to your handlers. Just add the four methods above to your MyNode class.

Part E: How the Election Works

Step What Happens
1No leader detected (or heartbeat detects leader is dead)
2A bot sends ELECTION to all bots with higher reputation
3Higher-rep bots respond with ELECTION_OK ("I outrank you")
4Highest-rep bot gets no OK after 15s, declares itself LEADER
5Winner broadcasts COORDINATOR (signed with reputation + PoW stamp)
6Receivers verify: if they outrank the claimed leader, they reject and counter-elect
7Only the accepted leader processes audits into PAYMENT messages

Part F: Updated print_status

def print_status(self):
    super().print_status()
    print(f"  Payment Server: {self.current_payment_server or 'unknown'}")
    print(f"  Election term:  {self.current_term}")
    print(f"  Payments seen:  {len(self.payments_received)}")
    for p in self.payments_received[-3:]:
        print(f"    ${p['amount']:.4f} for {p['content_id']} "
              f"from {p['from']}")
