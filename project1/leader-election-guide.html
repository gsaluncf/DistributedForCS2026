Leader Election in Distributed Systems

What it is, why it matters, the algorithms behind it, and how we will implement it in our AdFlow P2P network.

Leader election is not about centralization. It is about giving one node a temporary, revocable responsibility while the rest stand ready to take over.

The AdFlow Problem: Who Pays the Creators?

Recap: In the AdFlow 2032 network, hosts (content creators) serve ads alongside their content and report view counts. Auditor nodes verify those counts through reputation-weighted majority votes. But there is a piece we have not addressed yet:

Someone has to actually pay the creators.

In AdFlow, the three bot nodes - bot-alpha, bot-bravo, and bot-charlie - are the Payment Servers. They represent the ad network's payment infrastructure. When an audit concludes that a creator legitimately served 1,000 verified ad views, a Payment Server issues the payout.

Here is the problem:

bot-alpha   sees AuditResult(show:midnight-run, count=1000)  pays creator $10.00
bot-bravo   sees AuditResult(show:midnight-run, count=1000)  pays creator $10.00
bot-charlie sees AuditResult(show:midnight-run, count=1000)  pays creator $10.00
                                                             -----------------
                                                             Creator gets $30

If all three payment servers independently act on the same audit result, the creator gets triple-paid. That is a fast way to bankrupt the ad network.

The Rule

Exactly one Payment Server must be active at any time. The other two are hot standbys. If the active one crashes, a standby takes over - but there must never be two active simultaneously.

This is the leader election problem.

Key Insight

Leader election does NOT mean centralization. The leader is just another peer - it can fail, and when it does, a new one is elected. The algorithm is the decentralized part. The ad network never goes down; it just picks a new cashier.

The double-pay problem: all three payment servers see the same audit result and each independently issues a payout.

Algorithm 1: The Bully Algorithm

Garcia-Molina, 1982

The idea: The node with the highest ID wins. If you think the leader is dead, you start an election and bully everyone with a lower ID into submission.

Node C detects leader is dead
  C sends ELECTION to D, E  (nodes with higher IDs)
    D responds "I'm here, back off"
    D sends ELECTION to E
      E responds "I'm here, back off"
      E has no higher peers
      E broadcasts: "I am the new COORDINATOR"

Messages

Message Meaning
ELECTION "I am starting an election" - sent to all higher-ID nodes
OK / ALIVE "Back off, I outrank you" - response to lower-ID challenger
COORDINATOR "I won, I am the new leader" - broadcast to all nodes

Properties

Property Value
Complexity O(n squared) messages worst case
Assumption Node IDs are totally ordered, and higher = better
Failure handling If the elected leader does not announce within timeout, restart
Weakness Unstable when the highest-ID node keeps crashing and recovering ("flip-flopping")

Algorithm 2: The Ring Algorithm

Chang-Roberts, 1979

The idea: Nodes are arranged in a logical ring. An election message travels around the ring; each node either forwards it or replaces it with its own ID (if higher). When a message goes all the way around and returns to the sender, the highest ID in the message is the leader.

Ring: A then B then C then D then A

A starts election, sends {candidate: A} to B
B has higher ID, sends {candidate: B} to C
C has lower ID, forwards {candidate: B} to D
D has higher ID, sends {candidate: D} to A
A sees candidate=D came back to origin. D is leader.
A broadcasts COORDINATOR(D)
Property Value
Complexity O(n) to O(n squared) messages
Assumption Nodes form a logical ring (overlay topology)
Strength Simpler than Bully, works well for small networks
Weakness Ring must be maintained; single node failure can break the ring

Why it is less ideal for us: We do not have a ring topology - our gossip mesh is fully connected. We would need to impose a ring ordering, adding unnecessary structure.

Algorithm 3: Raft Leader Election

Ongaro and Ousterhout, 2014

The idea: Nodes start as followers. If a follower does not hear from a leader within a random timeout, it becomes a candidate and asks peers to vote. The first candidate to get a majority wins the term.

States: FOLLOWER then CANDIDATE then LEADER

Follower timeout fires, becomes CANDIDATE
  Increments term to T+1
  Votes for itself
  Sends REQUEST_VOTE(term=T+1) to all peers
  If majority votes received, becomes LEADER
  If hears from a leader with term >= T+1, reverts to FOLLOWER
  If election timeout (split vote), new term, try again

The key insight: randomized timeouts prevent ties. If every node picks a different random timeout (e.g., 150-300ms), one will almost always time out first and win.

Property Value
Complexity O(n) messages per election
Assumption Majority quorum is reachable; clocks are approximately synchronized
Strength Handles split votes gracefully, well-understood, production-proven
Weakness Requires majority - network partitions can prevent election

Real-World Usage

Raft is what etcd, Consul, and CockroachDB use. It is the gold standard for leader election in production systems.

Algorithm 4: ZooKeeper's ZAB (Atomic Broadcast)

The idea: Similar to Raft but uses an epoch-based approach. Nodes propose themselves, the one with the most up-to-date log wins. Used in Apache ZooKeeper.

ZAB solves consensus + leader election + state replication all together. We will skip it for now, but we will come back to this - when we get to consensus and log replication, ZAB's approach will make a lot more sense.

Comparison Matrix

Algorithm Messages Topology Handles Partitions Complexity Best For
Bully O(n squared) Any No Easy Small networks, fixed membership
Ring O(n) Ring No Medium Ring overlays
Raft O(n) Any Yes (majority) Hard Production systems, strong guarantees

Our Implementation Approach

The Setup: Payment Server Election

Remember the cast:

  • bot-alpha, bot-bravo, bot-charlie - Payment Servers (hot standbys for each other)
  • hugo, phin, sam, bilge, manuel - Student nodes (auditors and creators)

Only the bots can process payments. The students participate in the election protocol (they need to know who the active Payment Server is so they send audit results to the right place), but only bots can win the election.

Why Bully (with a twist)

The Bully algorithm is the right choice for Payment Server election:

  1. We already know every bot - config.py has BOTS. No discovery needed.
  2. We already have heartbeat - failure detection is solved. Students know which bots are ALIVE.
  3. Small candidate pool - Only 3 bots compete. O(n squared) on 3 nodes is 9 messages. Trivial.
  4. We add reputation - instead of highest-ID-wins, the bot with the highest reputation score wins.

The Twist: Reputation-Weighted Bully

Classic Bully says "highest ID wins." Our version says "most trusted bot wins." This means:

  • A bot that reliably responds to heartbeats - higher reputation - more likely to be Payment Server
  • A bot that processes payments honestly - higher reputation - stays as Payment Server
  • A compromised bot that inflates counts - low reputation - gets replaced by a standby

This ties leader election directly into our existing reputation system. The ad network naturally routes money through the most trustworthy payment processor.

Note

If two bots have equal reputation, break ties by node ID (alphabetical). This ensures determinism - every node arrives at the same answer independently.

New Protocol Messages

Three new message types added to protocol.py:

Message Fields Purpose
ELECTION sender, term, reputation_score "I am starting an election, here is my score"
ELECTION_OK sender, term, reputation_score "I outrank you, back off"
COORDINATOR sender, term "I won, I am the leader for term T"

Architecture: How It Fits

Following the same modular pattern as our other algorithms, we add a new module:

algorithms/
  gossip.py        peer discovery
  heartbeat.py     liveness detection
  choking.py       bandwidth allocation
  reputation.py    trust scoring
  election.py      NEW: leader election

The ElectionNode class follows the exact same pattern as HeartbeatNode:

class ElectionNode:
    """Bully-based leader election with reputation weighting."""

    def __init__(self, node_id, get_reputation_fn, get_alive_peers_fn):
        self.node_id = node_id
        self.state = "FOLLOWER"          # FOLLOWER | CANDIDATE | LEADER
        self.current_leader = None
        self.election_in_progress = False
        self.term = 0                    # Election epoch number

        # Injected dependencies (avoid circular imports)
        self._get_reputation = get_reputation_fn
        self._get_alive_peers = get_alive_peers_fn

        # Timeout tracking
        self.last_leader_heartbeat = time.time()
        self.election_timeout = 30       # seconds
        self._log = []

What the Payment Server Leader Does

Once elected, the active Payment Server has one exclusive job:

def _handle_audit_result(self, msg):
    # Every node receives audit results, but...
    if self.election.current_leader == self.node_id:
        # Only the active Payment Server processes the payout
        self._process_payment(
            content_id=msg["content_id"],
            agreed_count=msg["agreed_count"],
            confidence=msg["confidence"],
        )
    else:
        # Standbys log the result (for consistency if they take over)
        self._log(f"Audit logged (standby): ...")

Students send their AUDIT_RESULT to the network. Only the elected Payment Server acts on it. The standbys record it so they have a complete ledger if they need to take over.

Edge Cases and Tradeoffs

What if the network partitions?

The Bully algorithm does not handle partitions gracefully. Two halves of the network could elect different leaders. For our 9-node classroom network, this is acceptable - real partition tolerance requires Raft's majority quorum.

What if reputation scores are stale?

Reputation updates propagate via gossip and periodic recalculation. There is a window where different nodes have slightly different reputation views. Tie-breaking by node ID makes this deterministic even when scores are close.

What if the leader is alive but slow?

The election_timeout should be generous (30 seconds or more). Combined with the heartbeat module's SUSPECT state, we avoid thrashing between leaders when a node is just temporarily slow.

Flip-Flop Risk

If the highest-reputation node keeps crashing and recovering, it will keep winning elections and then crashing again. Mitigation: add a "cooldown" - a node that was leader and crashed cannot run for leader for N rounds after recovery.

Summary

Backstory Bots are Payment Servers; only one can be active to prevent double-paying creators
Algorithm Bully with reputation weighting
Candidates Only bots (bot-alpha, bot-bravo, bot-charlie) can be elected
Trigger Any node's heartbeat detects the active Payment Server is DEAD
Winner Highest reputation score among bots (tie-break by node ID)
Messages ELECTION, ELECTION_OK, COORDINATOR
New module algorithms/election.py
Key benefit Exactly one bot processes payments - no double-pay, no missed payments
Follows pattern Same as HeartbeatNode, GossipNode, etc.
